# Git Diff

Detect all git changes in the current repository, classify them using
repo-specific custom instructions, and produce a structured report that
downstream skills can consume.

## Arguments

- `base-ref` (optional) — git ref to diff against. Defaults to `HEAD`.
  Use `--staged` to limit to staged changes only.
- `-- <path>` (optional) — path filter passed to `git diff`.

Examples:
```
/git-diff                              → all uncommitted changes vs HEAD
/git-diff main                         → changes since diverging from main
/git-diff HEAD~3                       → last 3 commits
/git-diff --staged                     → staged changes only
/git-diff main -- src/components/      → scoped to a subdirectory
```

## Workflow

### 1. Gather Raw Changes

Run these git commands to collect all change information:

```bash
# Unstaged changes
git diff --name-status

# Staged changes
git diff --cached --name-status

# Untracked files
git ls-files --others --exclude-standard

# If base-ref provided, use it instead
git diff <base-ref> --name-status

# Summary stats
git diff [<base-ref>] --stat
```

Combine all results into a single deduplicated file list with statuses:
`A` (added), `M` (modified), `D` (deleted), `R` (renamed), `?` (untracked).

### 2. Read Repo Custom Instructions

Before classifying, look for a repo-level configuration file that tells the
skill how to interpret changes for this specific codebase. Search in order:

1. `.claude/git-diff.config.md` — dedicated config for this skill
2. `.claude/INSTRUCTIONS.md` — general Claude instructions (may contain
   git-diff hints)
3. `CLAUDE.md` at repo root — project-level instructions

If a config file is found, read it and use its rules to override or extend the
default classification below. The config can specify:

- **Custom categories** — project-specific buckets beyond the defaults
- **Path mappings** — which directories map to which categories
  (e.g. `micro-frontends/ep/src/ → EP Components`)
- **Story detection rules** — how to find whether a component has story coverage
  (e.g. co-located `*.stories.tsx`, or a separate `stories/` directory)
- **Key patterns to extract** — project-specific attributes to look for in
  changed files (e.g. `data-qa`, custom hooks, specific imports)
- **Downstream skill hints** — what skills to suggest for uncovered files

If no config is found, follow this fallback sequence:

1. **Inform** — Tell the user:
   > No `.claude/git-diff.config.md` found. Using inferred defaults.
   > For a tailored config, run `/codebase-analyzer`.

2. **Quick inference** — Run lightweight structural checks (do not do a full
   analysis — just enough to improve defaults):
   - **Monorepo detection**: check for `lerna.json`, `pnpm-workspace.yaml`,
     `nx.json`, `turbo.json`, multiple `package.json` files, or `packages/` /
     `micro-frontends/` directories.
   - **Story location pattern**: `find . -name "*.stories.*" -not -path
     "*/node_modules/*" -maxdepth 5 | head -10` — determine co-located vs
     separate `stories/` directory.
   - **Test naming pattern**: `find . \( -name "*.test.*" -o -name "*.spec.*"
     \) -not -path "*/node_modules/*" -maxdepth 5 | head -10` — determine
     dominant convention (`*.test.*` vs `*.spec.*`).
   - **Test ID attribute**: sample a few component files (`*.tsx`, `*.jsx`)
     and `grep -oh 'data-\(testid\|qa\|test\|cy\)'` to find the most common
     attribute.
   - **Data fetching library**: read the root `package.json` (and top-level
     sub-project `package.json` files if monorepo) for `@tanstack/react-query`,
     `react-query`, `swr`, `@apollo/client`, `@trpc/client` and note the
     version.

3. **Proceed with enhanced defaults** — Use the inferred patterns to modify
   the default category table at the start of Step 3 before classifying. For
   example:
   - If a monorepo is detected with `micro-frontends/ep/` and
     `micro-frontends/intg/`, replace the generic "Components" row with
     sub-project-scoped rows (e.g. "EP Components", "INTG Components").
   - If stories are in a separate `stories/` directory, update the Story
     Detection logic to search there instead of co-located paths.
   - If `*.spec.*` is the dominant test pattern, adjust the Tests category
     glob accordingly.
   - If a specific test ID attribute (e.g. `data-qa`) was found, use it
     instead of the default `data-testid`.

4. **Append tip to report** — At the very end of the output report (after
   "Suggested Actions"), add:
   ```
   ---
   **Tip:** This report used auto-detected defaults. For more accurate
   classification tailored to this repo, run `/codebase-analyzer` to generate
   a project-specific config.
   ```

### 3. Classify Changes

Assign each changed file to a category. Default categories (override via
repo config):

| Category | Default file patterns |
|----------|----------------------|
| Components | `*.tsx`, `*.jsx` under `src/components/`, `src/pages/`, `src/features/`, `src/views/` |
| Stories | `*.stories.tsx`, `*.stories.jsx`, `*.stories.ts` |
| Tests | `*.test.*`, `*.spec.*`, `__tests__/**` |
| Mocks | `**/mocks/**`, `**/__mocks__/**`, `*.mock.*` |
| Hooks/API | `**/hooks/**`, `**/api/**`, `**/services/**`, `**/*query*`, `**/*mutation*` |
| Styles | `*.css`, `*.scss`, `*.styled.*`, `*.styles.*` |
| Config | `package.json`, `tsconfig*`, `*config*`, `.storybook/**` |
| Docs | `*.md`, `*.mdx` |
| Other | everything else |

### 4. Analyze Changed Components

For each file in **Components** (added or modified):

1. Read the file
2. Extract: component name, props/interface, key imports, hooks used
3. Check if a corresponding story file exists (use repo config rules or
   search for `<ComponentName>.stories.*` co-located or in a stories dir)
4. Extract any test-id attributes (`data-testid`, `data-qa`, or per repo config)
5. Note API dependencies (query keys, fetch calls, service imports)

### 5. Produce Report

Output a structured markdown report:

```
## Git Diff Summary

**Base:** `<ref>`  |  **Branch:** `<branch>`  |  **Files changed:** N

### Change Breakdown

| Category | Count | Files |
|----------|-------|-------|
| Components | 3 | `Foo.tsx`, `Bar.tsx`, `Baz.tsx` |
| Stories | 1 | `Foo.stories.tsx` |
| ... | ... | ... |

### Components Missing Story Coverage

| Component | Path | Status | Props | Key Dependencies |
|-----------|------|--------|-------|------------------|
| Bar | `src/components/Bar.tsx` | Added | `{ items, onSelect }` | useQuery("bars") |
| Baz | `src/components/Baz.tsx` | Modified | `{ id, isDisabled }` | useFetch("/api/baz") |

### Detailed Change Analysis

#### `src/components/Bar.tsx` (Added)
- Props: `items: Item[]`, `onSelect: (id: string) => void`
- Hooks: `useQuery` with key `["bars"]`
- Test IDs: `bar-list`, `bar-item`
- Story: **None found** → candidate for generation

#### `src/components/Baz.tsx` (Modified)
- Added prop `isDisabled`
- Modified empty-state render path
- Story: `Baz.stories.tsx` exists → may need updating
```

### 6. Suggest Next Actions

Based on the analysis, output a numbered action list:

1. **Components without stories** → suggest running the appropriate generator
   skill (per repo config, or default to `/storybook-generator <path>`)
2. **Modified components with existing stories** → suggest reviewing/updating
   the story
3. **Changed mocks/hooks** → flag downstream stories that may be affected
4. **Deleted components** → flag orphaned stories referencing them

Format as a checklist the user can act on:

```
### Suggested Actions

- [ ] Generate story for `Bar.tsx` → `/storybook-generator src/components/Bar.tsx`
- [ ] Update story for `Baz.tsx` — prop `isDisabled` was added
- [ ] Review `Foo.stories.tsx` — mock handler `getFoos` was modified
- [ ] Delete orphaned story `Old.stories.tsx` — `Old.tsx` was removed
```

## Repo Config File Format

The config file (`.claude/git-diff.config.md`) is generated automatically by
`/codebase-analyzer`. It contains these sections — used by Steps 2–4 above:

- **Categories** — glob patterns mapping files to classification buckets
- **Story Detection** / **Test Detection** / **Impact Detection** — rules for
  finding coverage gaps or change impact (varies by purpose)
- **Extract Patterns** — attributes to look for in changed files (test IDs,
  query keys, hooks)
- **Downstream Skills** — what to suggest for uncovered files
- **Notes** — version differences, sub-project caveats

Run `/codebase-analyzer` to generate a config tailored to the repo. When no
config exists, the fallback in Step 2 provides inferred defaults.

## Edge Cases

- **No changes:** Report "Working tree clean" and exit.
- **Binary files:** List but skip content analysis.
- **Renamed files:** Show old → new path; check if stories reference the old name.
- **Deleted files:** Flag orphaned stories/tests that import the deleted file.
- **Merge conflicts:** Warn and skip conflicted files.
- **Monorepo:** If repo config defines sub-projects, scope analysis per project.
